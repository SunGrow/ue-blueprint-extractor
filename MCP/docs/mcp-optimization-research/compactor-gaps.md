# Compactor Coverage Analysis

## Task 1: What the Compactor Currently Handles

**File:** `MCP/src/compactor.ts` (150 lines)

### Data structures compacted

1. **Blueprint graphs** (`bp.functions[]`) -- the primary target
2. **Widget trees** (`bp.widgetTree.rootWidget`) -- secondary target

### Entry point

`compactBlueprint(data)` expects a parsed JSON object with a top-level `blueprint` key.
It dispatches into two independent branches:

- **Branch 1 -- `compactGraph(graph)`** for each entry in `blueprint.functions[]`
- **Branch 2 -- `compactWidgetNode(node)`** for `blueprint.widgetTree.rootWidget`

### Fields stripped / transformed

#### Graph-level (`compactGraph`)

| Field | Action |
|-------|--------|
| `graph.graphGuid` | Deleted |
| `node.nodeGuid` | Deleted; replaced with sequential `node.id` = `n0`, `n1`, ... |
| `node.posX` | Deleted |
| `node.posY` | Deleted |
| `node.nodeComment` | Deleted when empty string |

#### Pin-level (`compactPin`)

| Field | Action |
|-------|--------|
| `pin.pinId` | Deleted |
| `pin.autogeneratedDefaultValue` | Deleted |
| `pin.defaultValue` | Deleted when empty string |
| `pin.type.sub_category` | Deleted when empty string |
| `pin.type` (whole object) | Replaced with string `"exec"` when `type.category === "exec"` |
| `pin.connections` | Deleted when empty array |
| `pin.connections[].nodeGuid` | Rewritten from full GUID to short ID (`n0`, `n1`, ...) |

#### Widget tree (`compactWidgetNode`)

| Field | Action |
|-------|--------|
| `node.displayLabel` | Deleted when equals `node.name` or is empty |
| `node.visibility` | Deleted when `"Visible"` (the default) |
| `node.properties` | Deleted when empty object `{}` |
| Children | Recursively compacted |

### Summary

The compactor is narrowly scoped: it only handles Blueprint graph nodes/pins and widget tree hierarchies. It achieves ~50-70% reduction on Full-scope Blueprint extractions. It does not touch any other asset type.

## Task 3: Low-Value Fields in Non-Compacted Asset Types

### extract_statetree (Priority: HIGH)

StateTree JSON typically 10-100KB. Expected low-value fields based on UE StateTree serialization:

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `stateGuid`, `taskGuid`, `transitionGuid` | GUID | Internal editor identifiers, no semantic value for LLM |
| `nodeIndex`, `dataIndex` | Internal indices | Runtime binding handles, not meaningful for understanding logic |
| `instanceDataOffset`, `instanceDataSize` | Runtime layout | Memory layout metadata irrelevant for reasoning |
| Empty `conditions[]`, empty `transitions[]` | Empty defaults | Noise when a state has no conditions/transitions |
| `schemaVersion`, `editorData` | Metadata | Editor-only serialization versioning |

**Estimated reduction: 20-35%**

### extract_behavior_tree (Priority: HIGH)

BT JSON includes full node hierarchy. Expected low-value fields:

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `nodeGuid`, `treeAssetGuid` | GUID | Editor identifiers |
| `posX`, `posY` | Position | Node layout in BT editor graph |
| `nodeInstanceMemorySize`, `instanceMemorySize` | Runtime layout | Memory allocation sizes |
| `executionIndex` | Runtime ordering | Internal BT execution order index |
| `childIndex`, `parentIndex` | Redundant tree | Indices redundant with the nested hierarchy |
| Empty `services[]`, empty `decorators[]` | Empty defaults | Noise for leaf nodes |

**Estimated reduction: 25-40%**

### extract_widget_blueprint (Priority: HIGH)

This is a dedicated widget-focused extractor returning widget tree + bindings + animations + compile status. The existing compactor already has `compactWidgetNode()` but it is NEVER called from this path.

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `displayLabel` (when == name) | Redundant | Same as `compactWidgetNode` already strips |
| `visibility` (when "Visible") | Default value | Same as `compactWidgetNode` already strips |
| Empty `properties {}` | Empty default | Same as `compactWidgetNode` already strips |
| `slotGuid`, `widgetGuid` | GUID | Internal editor identifiers |
| Blueprint graph data in widget BP | Graph data | Same fields as `compactGraph` strips |

**Estimated reduction: 15-30%** (the C++ extractor may already be fairly compact, but widget trees with many children accumulate redundant defaults)

### extract_material / extract_material_function (Priority: MEDIUM)

Material graph snapshots. These already have a `verbose` flag controlling C++-side detail.

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `materialExpressionGuid` | GUID | Internal expression identifiers |
| `posX`, `posY` (expression positions) | Position | Layout in material editor |
| `sortPriority` | Metadata | Editor rendering order |
| Default parameter values matching CDO | CDO defaults | Properties at default values add no information |
| `editorComments[]` when empty | Empty default | Noise |

**Estimated reduction: 15-25%** (lower because `verbose=false` already suppresses some detail)

### extract_material_instance (Priority: MEDIUM)

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `parameterGuid` | GUID | Per-parameter identifiers |
| `parameterGroup` when "None" | Default value | Default group has no information |
| `overrideType` internal enum values | Metadata | Runtime override type flags |
| Empty parameter arrays | Empty default | `fontParameters: []` etc. are noise |

**Estimated reduction: 10-20%**

### extract_datatable (Priority: MEDIUM)

DataTables can be very large (200KB+ triggers truncation already).

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| Per-row GUIDs (if serialized) | GUID | Row identification GUIDs |
| Struct field type metadata repeated per row | Redundant schema | Type info in schema is sufficient, per-row repetition is waste |
| Default/zero values for numeric fields | CDO defaults | Rows with all-default fields in optional columns |

**Estimated reduction: 10-30%** (varies heavily by table structure; could be dramatic for wide sparse tables)

### extract_dataasset (Priority: LOW)

DataAssets are typically small, user-defined property bags.

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| Property type metadata | Redundant | Type info may duplicate what the class definition provides |
| GUIDs for struct/object references | GUID | Internal reference GUIDs |

**Estimated reduction: 5-15%**

### extract_widget_animation (Priority: MEDIUM)

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| Section GUIDs / binding GUIDs | GUID | Internal Sequencer identifiers |
| `sortOrder` | Metadata | Track display order in editor |
| Default interpolation modes | Default value | When all keys use the same default interp mode |

**Estimated reduction: 10-20%**

### extract_blackboard / extract_user_defined_struct / extract_user_defined_enum (Priority: LOW)

These are typically small schema-like responses (key lists, field lists, enum entries). Compaction would save very few tokens.

**Estimated reduction: <10%**

### extract_curve / extract_curvetable (Priority: LOW)

Curve data is mostly numeric keyframe values (time, value, tangent). Little to strip without losing data.

**Estimated reduction: <10%**

### extract_anim_sequence / extract_anim_montage / extract_blend_space (Priority: LOW)

Animation metadata extractions. Typically compact already (notifies, sections, samples).

| Field pattern | Category | Rationale |
|---------------|----------|-----------|
| `notifyGuid` | GUID | Notify instance identifiers |
| `segmentGuid` | GUID | Montage segment identifiers |
| Default trigger weights / timing | Default value | When at engine defaults |

**Estimated reduction: 5-15%**

## Task 4: Compactor Extension Recommendations

### Priority: HIGH

#### 1. Wire `extract_widget_blueprint` through existing compactor

**Effort:** Minimal -- the `compactWidgetNode()` and `compactGraph()` functions already exist.

The `extract_widget_blueprint` tool calls `ExtractWidgetBlueprint` which returns a response that likely contains `blueprint.widgetTree` and possibly `blueprint.functions`. The existing compactor would handle both, but it is never invoked. Adding `compact` param and calling `compactBlueprint(parsed)` would give immediate gains.

- Fields stripped: displayLabel (redundant), visibility (default), empty properties, graph GUIDs, posX/posY, pin noise
- Estimated reduction: 15-30%
- Implementation: Add `compact` boolean param, call `compactBlueprint(parsed)` (same as `extract_blueprint`)

#### 2. `extract_behavior_tree` compactor

**Effort:** New compactor function needed.

BehaviorTree responses contain a node hierarchy with GUIDs, position data, and runtime memory sizing. A `compactBehaviorTree(data)` function should:

- Strip `nodeGuid`, `treeAssetGuid`
- Strip `posX`, `posY`
- Strip `instanceMemorySize`, `nodeInstanceMemorySize`, `executionIndex`
- Remove empty `services[]` and `decorators[]` arrays
- Remove empty/default string fields

- Estimated reduction: 25-40%
- Implementation: ~40-60 lines, new `compactBehaviorTree()` function

#### 3. `extract_statetree` compactor

**Effort:** New compactor function needed.

StateTree responses include GUIDs, data layout offsets, and editor metadata. A `compactStateTree(data)` function should:

- Strip all `*Guid` fields (stateGuid, taskGuid, transitionGuid)
- Strip `instanceDataOffset`, `instanceDataSize`, `dataIndex` runtime layout fields
- Remove empty `conditions[]` and `transitions[]` arrays
- Strip `editorData` / `schemaVersion` if present

- Estimated reduction: 20-35%
- Implementation: ~30-50 lines, new `compactStateTree()` function

### Priority: MEDIUM

#### 4. `extract_material` / `extract_material_function` compactor

**Effort:** New compactor function OR expand C++-side `verbose` flag.

Material graphs have the same posX/posY + GUID pattern as Blueprint graphs. Since `verbose=false` already strips some detail, this is partially covered. A compactor could:

- Strip expression GUIDs, posX/posY
- Remove empty comment arrays
- Strip default sort priorities

- Estimated reduction: 15-25% (on top of verbose=false)
- Implementation: ~25-40 lines, or extend C++ verbose mode

#### 5. `extract_datatable` compactor

**Effort:** Medium -- needs understanding of per-row structure.

DataTables can exceed 200KB. Compaction should:

- Strip per-row GUIDs if present
- Deduplicate type schema (emit once, not per row)
- Strip fields at default/zero values (aggressive mode)

- Estimated reduction: 10-30% (highly variable)
- Implementation: ~30-50 lines depending on row structure variability

#### 6. `extract_material_instance` compactor

- Strip parameterGuid fields
- Remove empty parameter arrays (fontParameters, runtimeVirtualTextureParameters when empty)
- Strip "None" parameterGroup

- Estimated reduction: 10-20%
- Implementation: ~20-30 lines

#### 7. `extract_widget_animation` compactor

- Strip section/binding GUIDs
- Remove default sortOrder values
- Remove default interpolation modes

- Estimated reduction: 10-20%
- Implementation: ~15-25 lines

### Priority: LOW

Blackboard, UserDefinedStruct, UserDefinedEnum, Curve, CurveTable, AnimSequence, AnimMontage, BlendSpace -- these are typically small responses where compaction saves <10%. Not worth dedicated compactor code unless profiling shows specific large cases.

## Architecture Recommendation

Rather than adding N separate compactor functions, consider a generic field-stripping utility:

```
function stripFields(obj, patterns: string[]): void
```

Common patterns across all asset types:
- `*Guid` / `*guid` fields (universal low-value for LLM)
- `posX`, `posY` (universal layout noise)
- Empty strings, empty arrays, empty objects
- Known default values per type

This would allow each tool to declare its strip-list without duplicating traversal logic. The existing `compactBlueprint` is too specialized (short-ID rewriting, exec pin simplification) to generalize fully, but the field-stripping portion could be shared.

### Quick-win execution order

1. **Wire `extract_widget_blueprint` through existing `compactBlueprint()`** -- zero new code in compactor, just add param + call in handler
2. **Add `compactBehaviorTree()`** -- high usage tool, 25-40% savings
3. **Add `compactStateTree()`** -- high usage tool, 20-35% savings
4. **Generic GUID/position stripper** -- medium effort, covers remaining tools
