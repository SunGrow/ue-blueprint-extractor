## 2. Detailed Audit Findings

### 2.1 Runtime Optimizations (P3, P4, P7, dual serialization)

#### P3 -- checkConnection TTL cache: IMPLEMENTED

- `connectionCacheTtlMs` defaults to 2000ms (`ue-client.ts:6`)
- Cache fields: `lastConnectionStatus`, `lastConnectionCheckAt` (`ue-client.ts:58-59`)
- Cache hit logic returns early when TTL hasn't expired (`ue-client.ts:103-107`)
- Concurrent dedup via `pendingConnectionCheck` (`ue-client.ts:109-111`)
- Invalidation on failure path (`ue-client.ts:84-87`, called at line 256)
- **Tests:** `ue-client.test.ts:53-94` covers cache hit, cache miss (TTL expiry), and concurrent dedup

#### P4 -- JSON parse/stringify chain reduction: PARTIAL

- 83 total `JSON.parse`/`JSON.stringify` occurrences across 23 files under `MCP/src/`
- No explicit parse-then-stringify roundtrips found in tool handlers
- **Remaining issue:** `normalizeToolSuccess` (`tool-results.ts:168-174`) still `JSON.stringify`s the envelope for the text content block even though the same object is also passed as `structuredContent`. This is the dual serialization problem (see below)
- Tool handlers stringify payloads for subsystem RPCs -- this is intentional, not redundant

#### P7 -- runs Map LRU cleanup: IMPLEMENTED

- `DEFAULT_MAX_RUN_HISTORY = 50` (`automation-controller.ts:100`)
- `pruneRuns()` method at lines 451-480:
  - Age-based cleanup (lines 460-463): deletes terminal runs older than `terminalRunRetentionMs` (default 24h)
  - Count-based cleanup (lines 469-479): keeps only top `maxRunHistory` terminal runs sorted by time
- Cleanup triggered before operations at lines 287, 350, 381, 387, 407

#### Dual text+structuredContent: NOT IMPLEMENTED (optimization not applied)

- `normalizeToolSuccess` (`tool-results.ts:168-174`):
  ```typescript
  return {
    content: [
      { type: 'text', text: JSON.stringify(envelope) },  // serialization #1
      ...extraContent,
    ],
    structuredContent: envelope,                          // serialization #2 on wire
  };
  ```
- Same pattern in `normalizeToolError` (lines 141-145)
- Every tool response is serialized to JSON twice -- once explicitly for the text block, once implicitly when MCP SDK transmits `structuredContent`

### 2.4 Compactor Implementations (P5, P9)

#### P5a -- extract_widget_blueprint through compactor: IMPLEMENTED

- `compactWidgetBlueprint()` at `compactor.ts:57-84`
- Strips: redundant `displayLabel`, default `visibility`, empty `properties`, empty `bindings`/`animations`/`compile` fields
- Recurses through `rootWidget` and `functions` (graph compaction)
- Wired at `widget-extraction.ts:3` (import) and `:60` (called when `compact=true`)

#### P5b -- compactBehaviorTree(): IMPLEMENTED

- `compactBehaviorTree()` at `compactor.ts:86-102`
- Strips: `schemaVersion`, `nodeIndex`, `executionIndex`, empty `properties`, empty `decoratorLogic`
- Recurses through `services`, `decorators`, `children`
- Wired at `extraction.ts:4` (import) and `:178-179` (called for `asset_type='behavior_tree'` when `compact=true`)

#### P5c -- compactStateTree(): IMPLEMENTED

- `compactStateTree()` at `compactor.ts:104-128`
- Strips: `schemaVersion`, `expressionIndent=0`, empty `nodeProperties`/`instanceProperties`
- Handles: `evaluators`, `globalTasks`, `states` (recursive), `transitions` with conditions
- Wired at `extraction.ts:6` (import) and `:180` (called for `asset_type='statetree'` when `compact=true`)

#### P9 -- Generic GUID/position stripper: NOT IMPLEMENTED

- No `stripFields(obj, patterns)` utility found anywhere in `MCP/src/`
- `posX`/`posY` deletion only exists inside `compactGraph()` at `compactor.ts:155-156` (Blueprint-specific)
- No pattern-based field removal (e.g., `/.*Guid$/`, `/^pos[XY]$/`) for arbitrary objects
- SUMMARY.md recommends: "Implement a generic `stripFields(obj, patterns)` utility for the common GUID/position/empty-default pattern"

#### Material graph compactor: NOT IMPLEMENTED

- `extract_material` (`extraction.ts:217-227`) returns raw subsystem results with no compaction
- `extract_material_function` (`extraction.ts:245-255`) same -- no compaction
- No `compactMaterial()` function in `compactor.ts`
- The word "compact" in extract_material description refers to server-side extraction format, not client-side compaction
- SUMMARY.md estimates 15-25% reduction via posX/posY + GUID stripping for material results

## 6. Overall Assessment

**Project health: Good.** The MCP server has successfully completed its major architectural transformation from a 9,340-line monolith to a clean 47-file modular architecture. 8 of 14 optimization items are fully implemented (57%), with the remaining work being polish and coverage.

The single critical defect (dual serialization) is a straightforward fix with clear options. The test coverage gap is the largest debt item but does not affect runtime correctness -- `server-contract.test.ts` (129 KB) provides substantial integration coverage as a safety net.
